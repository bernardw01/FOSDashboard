# Feature: Agreement Status Updates on Delivery P&L

> **PRD version 2.12.2** - **Status: Shipped** (v2.12.0; create fixes **v2.12.1** enum id, **v2.12.2** document write). Extends feature **006** (`#panel-delivery`, per-project P&L). Fibery type: **`Agreement Management/Status Updates`**. See [018-agreement-status-updates-delivery-pnl-implementation-plan.md](018-agreement-status-updates-delivery-pnl-implementation-plan.md).

---

## Goal

Let any user who can open the **Delivery** dashboard (**Projects & P&L**, route `delivery`) **read the latest agreement status update** for a selected project and **submit a new update** to Fibery without leaving the P&L card.

When a project row is selected, the P&L fetch MUST include that agreement's status-update history (at least the **latest** row). The UI MUST show a **traffic-light status** (Red / Yellow / Green), a short summary of the latest update, and an **Add status update** button that opens a modal to compose rich text and pick the current status. On submit, the server MUST set **Submitted by** from the signed-in user's Workspace email (never from client input).

**Primary audience:** Delivery leads, project managers, and finance partners running weekly project reviews.

**Non-goals (v1):**

- Editing or deleting existing status updates from the dashboard.
- Email or Slack notifications on new updates.
- Surfacing status updates on Agreement, Revenue review, or Utilization panels.
- A separate Drive snapshot artifact (status rows ship **inside** each `delivery-pnl/<id>.json`).
- Replacing Fibery as the system of record for status narrative.

## Data source review

**Intent:** Read and create rows in Fibery **`Agreement Management/Status Updates`**, linked to **`Agreement Management/Agreements`**, through the existing server-side **`fiberyClient.js`** path (same token and batch-command pattern as AI usage upserts). The browser MUST NOT call Fibery directly.

| Canonical field | Fibery field | Type | Use |

## User stories

- As a **delivery lead**, I want to see the **latest Red / Yellow / Green status** for the project I selected in the P&L card so I know the current delivery health narrative without opening Fibery.
- As a **project manager**, I want to **add a status update** (rich text + status) from the P&L card so my weekly note is captured in the same system delivery leadership already uses.
- As a **finance reviewer**, I want **Submitted by** and **date** on the latest update visible next to the margin KPI so I can trust who last attested to status.
- As an **authorized user**, I want status updates **loaded with the monthly P&L** for the selected agreement so I do not wait on a second manual refresh.
- As a **historical reviewer**, I want status updates **included in daily snapshots** so I can see the latest attested status when viewing a past snapshot date (read-only).
- As **security / ops**, I want create operations to use the **server Fibery token** and **spreadsheet auth** only; the client must not send `Submitted by` as a trusted field.

## Acceptance Criteria (testable)

- [ ] **Access gate.** Users who can call `getDeliveryProjectMonthlyPnL(agreementId)` (spreadsheet-authorized via `requireAuthForApi_()`) can **read** status updates and **submit** new ones in **Live data** mode. Users who fail auth receive the same safe errors as other Delivery endpoints (no stack traces).
- [ ] **P&L payload extension.** `getDeliveryProjectMonthlyPnL(agreementId)` returns a `statusUpdates` block: `{ latest: ?Object, history: !Array<!Object>, statusOptions: !Array<!Object> }`. `latest` is the most recent row by `fibery/creation-date` for that agreement, or `null` when none exist. Bump **`DELIVERY_PNL_CACHE_SCHEMA_VERSION_`** to **5** (client suffix `_v5`).
- [ ] **Snapshot inclusion.** The daily snapshot job MUST persist `statusUpdates` inside each **`delivery-pnl/<agreementId>.json`** because it calls the same **`buildDeliveryProjectMonthlyPnLInternal_`** builder. **`SNAPSHOT_EXPECTED_SCHEMA_VERSIONS_['delivery-pnl']`** MUST be **5** after release. Legacy snapshot dates with schema **4** remain loadable without status data.
- [ ] **Latest status chip.** When a project is selected and P&L data is loaded, a **status summary card** renders **to the right of the Margin KPI chip** in `#delivery-pnl-kpi-strip`. It shows: traffic-light dot + enum label (or "No status yet"), **Submitted by** + short date, and a **plain-text excerpt** of `Update`.
- [ ] **Add flow (live only).** **Add status update** opens `#deliveryStatusUpdateModal`. Fields: **Status** (required select mapping to the three enum names, including **`Agreement Off Trajectory`**) and **Update** (required rich-text editor). Submit calls `createAgreementStatusUpdate(...)`. **Disabled or hidden in snapshot data-source mode.**
- [ ] **Server create.** `createAgreementStatusUpdate` sets Agreement relation, enum, **Submitted by** from server session, and Update document from sanitized content. On success, client invalidates per-project P&L cache and refetches.
- [ ] **No P&L regression.** Monthly grid, chart, pacing strip, and existing KPI chips behave as before when no status updates exist.
- [ ] **Activity logging.** Whitelist: `delivery_status_update_modal_open`, `delivery_status_update_submit` (Route `delivery`).
- [ ] **PRD / version discipline (on ship).** MINOR → **2.12.0**; **FR-112** + **AC-70**; extend **FR-104** / **AC-60** / **AC-61** for P&L schema 5; full version sweep.

## Data Model

### Normalized status update row

```javascript
{
  id: string,
  agreementId: string,
  agreementStatus: string,      // enum/name
  trafficLight: 'green'|'yellow'|'red',
  submittedBy: string,
  createdAt: string,            // ISO
  updatePlain: string
}
```

### P&L payload extension (`cacheSchemaVersion: 5`)

```javascript
{
  // ... existing months, lifetime, laborRoles, etc.
  statusUpdates: {
    latest: ?NormalizedStatusUpdate,
    history: NormalizedStatusUpdate[],
    statusOptions: [
      { key: 'on_track', label: 'Green - On track', enumName: 'Agreement On Track', trafficLight: 'green' },
      { key: 'at_risk', label: 'Yellow - At risk', enumName: 'Agreement At Risk', trafficLight: 'yellow' },
      { key: 'off_trajectory', label: 'Red - Off trajectory', enumName: 'Agreement Off Trajectory', trafficLight: 'red' }
    ]
  }
}
```

### Script Properties (`adminSettingsRegistry.js`)

| Property | Default | Purpose |
| --- | --- | --- |
| `DELIVERY_STATUS_UPDATES_MAX_ROWS` | `20` | Max history rows per P&L fetch |
| `DELIVERY_STATUS_UPDATE_MAX_CHARS` | `8000` | Server cap on update body length |

### Cache / snapshots

| Mode | Behavior |
| --- | --- |
| **Live** | `sessionStorage` `fos_delivery_pnl_<id>_v5` includes `statusUpdates`. |
| **Snapshot** | `getDashboardSnapshotPnl` returns the same shape from Drive; **no Fibery fetch**; submit disabled. |
| **Legacy P&L v4 snapshots** | P&L loads; chip shows inline message that status updates were not captured. |

## Operations

### Queries

- **`fetchStatusUpdatesForAgreement_(agreementId, limit)`** - filter by Agreement id, `creation-date` desc.
- Invoked from **`buildDeliveryProjectMonthlyPnLInternal_`** (live API + snapshot job).

### Actions

| API | Auth | Behavior |
| --- | --- | --- |
| `getDeliveryProjectMonthlyPnL(agreementId)` | `requireAuthForApi_()` | Read (includes statusUpdates) |
| `createAgreementStatusUpdate(...)` | `requireAuthForApi_()` | Fibery create (live only) |

**Create entity (illustrative):**

```javascript
{
  'Agreement Management/Agreement': { 'fibery/id': agreementId },
  'Agreement Management/Agreement Status': 'Agreement Off Trajectory', // enum/name
  'Agreement Management/Submitted by': userEmail,
  'Agreement Management/Update': { 'fibery/document-content': sanitizedContent }
}
```
